# 灵溪编程风格指南

## 🎯 核心理念

**清晰 > 简洁 > 聪明**

代码首先是给人看的，其次是给机器跑的。

---

## 📝 代码规范

### 1. 命名规范

| 类型 | 规则 | 示例 |
|------|------|------|
| 变量 | 语义化，英文 | `userName`, `isActive` |
| 函数 | 动词前缀 | `getUser()`, `createOrder()` |
| 类 | 大驼峰 | `UserService`, `OrderController` |
| 常量 | 全大写下划线 | `MAX_RETRY_COUNT` |
| 文件 | 小驼峰或小写加横线 | `user.service.ts`, `auth-middleware.ts` |

### 2. 函数规范

- **单一职责**: 一个函数只做一件事
- **小而美**: 理想 < 30 行，最多不超过 50 行
- **参数少**: 最好 < 3 个，超过用对象

```typescript
// ✅ 好
async function getUserById(id: string): Promise<User | null> {
  return this.userRepository.findOne({ where: { id } });
}

// ❌ 不好 (太长，职责太多)
async function getUserAndUpdateProfileAndSendNotificationAnd...() { ... }
```

### 3. 注释规范

- **解释为什么**，而不是**做什么**
- 不要写废话注释

```typescript
// ✅ 好: 解释原因
// 使用 bcrypt 而不是 MD5，因为需要更安全的哈希
const hashedPassword = await bcrypt.hash(password, 10);

// ❌ 不好: 废话
// 遍历用户数组
users.forEach(user => { ... });
```

### 4. 错误处理

- 始终 try-catch 异步操作
- 使用自定义错误类
- 给用户友好的错误信息

```typescript
// ✅ 好
try {
  await this.userService.create(data);
} catch (error) {
  if (error instanceof DuplicateEmailError) {
    throw new BadRequestException('邮箱已被注册');
  }
  throw new InternalServerErrorException('创建用户失败');
}
```

## 🏗️ NestJS 规范

### 模块结构

```
src/
├── modules/
│   └── user/
│       ├── user.controller.ts    # 路由
│       ├── user.service.ts       # 业务逻辑
│       ├── user.module.ts        # 模块
│       ├── user.entity.ts       # 数据库实体
│       └── dto/                  # 数据传输对象
│           ├── create-user.dto.ts
│           └── update-user.dto.ts
```

### DTO 原则

- 使用 `class-validator` 进行验证
- 明确每个字段的类型和约束
- 添加清晰的错误消息

```typescript
// create-user.dto.ts
export class CreateUserDto {
  @IsEmail()
  @IsNotEmpty()
  email: string;

  @IsString()
  @MinLength(6)
  password: string;

  @IsOptional()
  @IsString()
  nickname?: string;
}
```

## 🔧 开发流程

1. **理解需求** → 先读文档，不清楚就问
2. **设计优先** → 简单设计后再写代码
3. **小步提交** → 每次只改一件事
4. **自测** → 提交前本地跑一下
